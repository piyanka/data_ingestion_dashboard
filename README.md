# Breathe ESG 

<img width="2934" height="1678" alt="image" src="https://github.com/user-attachments/assets/aec5ca51-9ddb-4f22-bada-f51ecc8d3570" />



A realistic ESG data ingestion prototype for the Breathe ESG assignment.

The app focuses on the parts that matter for the brief:

- raw source preservation
- normalization into a canonical activity model
- validation and suspicious-row review
- analyst approval before rows are locked
- auditability and source traceability

## Stack

- Backend: Django + Django REST Framework
- Frontend: React + Vite
- Database: SQLite for local development, Postgres via `DATABASE_URL` in production
- Deployment target: Render or Railway

## Local Development

### 1) Backend

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Set the environment variables for local development:

```bash
export DJANGO_SECRET_KEY="dev-secret-key"
export DJANGO_DEBUG=1
export DJANGO_ALLOWED_HOSTS="localhost,127.0.0.1"
export CORS_ALLOWED_ORIGINS="http://localhost:5173,http://127.0.0.1:5173"
```

Run migrations, bootstrap the admin user if needed, and start the server:

```bash
python manage.py makemigrations
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver 127.0.0.1:8000
```

If you want a local admin bootstrap that behaves like production, set these first:

```bash
export DJANGO_ADMIN_USERNAME="admin"
export DJANGO_ADMIN_EMAIL="admin@example.com"
export DJANGO_ADMIN_PASSWORD="change-me"
python manage.py bootstrap_admin
```

### 2) Frontend

```bash
cd frontend
npm install
```

Create a local Vite env file and point it at the backend:

```bash
cp .env.example .env.local
```

Start the frontend:

```bash
npm run dev
```

The frontend will usually run on `http://127.0.0.1:5173` and the backend on `http://127.0.0.1:8000`.

## Production Deployment

### Backend on Render

This repository includes a Render Blueprint at [render.yaml](/Users/priyanka/Documents/Projects/breathe_esg/render.yaml) and a backend build script at [backend/build.sh](/Users/priyanka/Documents/Projects/breathe_esg/backend/build.sh).

The repo root also includes [.python-version](/Users/priyanka/Documents/Projects/breathe_esg/.python-version) so Render uses Python 3.12 instead of the default 3.14 runtime. Django 4.2 officially supports Python 3.12, so this avoids the admin and template errors you were seeing.

Render settings used by the Blueprint:

- `DATABASE_URL` from the managed Render Postgres database
- `DJANGO_SECRET_KEY` generated by Render
- `DJANGO_DEBUG=0`
- `DJANGO_ALLOWED_HOSTS`
- `CORS_ALLOWED_ORIGINS`

The backend service build flow is:

```bash
./build.sh
gunicorn config.wsgi:application --bind 0.0.0.0:$PORT
```

Note: the free Render tier does not support `preDeployCommand`, so migrations run inside `backend/build.sh`.

### Frontend on Render

Deploy the React app as a separate Render Static Site:

- Build command: `npm install && npm run build`
- Root directory: `frontend`
- Publish directory: `dist`
- Environment variable: `VITE_API_BASE_URL=https://<your-backend-service>.onrender.com/api/v1`

Important: the frontend will fail in production if `VITE_API_BASE_URL` is not set, because it will fall back to localhost.

### Suggested Render flow

1. Push this repo to GitHub.
2. In Render, create a new Blueprint from `render.yaml`.
3. Let Render create the PostgreSQL database and backend web service.
4. Copy the backend service URL from Render.
5. Create a new Static Site for `frontend/`.
6. Set `VITE_API_BASE_URL` to the deployed backend URL.
7. Build and deploy the frontend static site.

To avoid Render shell access, add these backend service environment variables in Render:

- `DJANGO_ADMIN_USERNAME`
- `DJANGO_ADMIN_EMAIL`
- `DJANGO_ADMIN_PASSWORD`

The backend build script runs `python manage.py bootstrap_admin` after migrations, so the admin user is created or refreshed automatically during deploy.

## Why Production Can Fail

The most common production problems for this app are configuration issues, not code bugs:

1. `VITE_API_BASE_URL` is missing, so the React app falls back to localhost and API calls fail in production.
2. `DJANGO_ALLOWED_HOSTS` is wrong. Django expects hostnames, not full URLs.
3. `CORS_ALLOWED_ORIGINS` does not include the deployed frontend origin, so browser requests are blocked.
4. `DATABASE_URL` is missing, so the backend uses SQLite instead of Postgres.
5. `collectstatic` was not run, so Django admin/static assets do not load properly.
6. `DJANGO_DEBUG` is left on in production.

## Notes

- The prototype keeps calculations intentionally simple.
- The point of the project is lineage, normalization, validation, and review workflow, not complex emissions science.
- For deployment, prefer Postgres in production and SQLite only for local development.
